# Testing Rules

> This document covers only "judgment-required rules" that ESLint/Prettier cannot enforce automatically.
> Mechanical rules are applied by `init-coding-rules` through conversational diff + approval based on this coding-rules folder.

---

## 테스트 철학

### E2E — Wide but Shallow (전체 플로우, 얕은 검증)
- **범위:** 사용자가 경험하는 전체 플로우 커버 (넓게)
- **깊이:** 내부 로직/구현 세부사항은 검증하지 않음 (얕게)
- **관점:** "이 플로우가 사용자 입장에서 작동하는가"
- **예:** 로그인 성공/실패/로그아웃 플로우 ✓ / 토큰 만료 내부 로직 ✗ (→ unit)
- E2E는 Playwright Test Agents가 자동 생성 (수동 작성/편집 금지)

### Unit / Logic Test — 요구사항 완전 커버 (깐깐한 검증)
- **목적:** plan.md 요구사항이 빠짐없이 구현됐는지 검증
- **기준:** plan.md 요구사항 n개 → 테스트 케이스 n개 이상 (1:1 매핑)
- **원칙:** "통과"가 목적이 아님. "요구사항 완전 반영" 확인이 목적.
- **방식:** plan.md 요구사항 목록을 체크리스트 삼되, 각 항목을 구현에 필요한 로직 계약으로 확장해 테스트 작성
- **확장 범위:** 정상 동작뿐 아니라 실패, 경계, 예외, 복구 가능성까지 포함해 로직을 명세

### Contract-first 원칙
- 테스트는 구현 검증 이전에 요구사항 계약(Contract)을 먼저 고정하는 문서다.
- plan.md의 제약/요구사항 ID를 테스트명에 연결해 추적 가능성을 유지한다.
- 구현은 테스트 계약을 만족하도록 뒤따라와야 한다.

## E2E-First Strategy

E2E tests serve as the safety net for AI-driven development. Full user flows are verified with E2E tests, and isolated logic is verified with stack-appropriate logic tests. E2E tests are automatically generated and repaired by Playwright Test Agents; logic tests are generated by plan-tests during planning and executed by developers.

---

## E2E-First TDD Workflow

### Features with UI

```
plan (+ plan-tests: Logic Tests) → publisher (UI) → Playwright Planner (specs) → Generator (E2E)
                                     → developer (copies tests → Red → implements → Green) → E2E pass
```

### Backend API Only

```
plan (+ plan-tests: Logic Tests) → backend-developer (copies tests → Red → implements → Green)
```

### On UI Changes

```
publisher (UI update) → Healer (E2E auto-repair)
→ developer (updates implementation) → Logic test pass → E2E pass
```

**Core principles:**
- E2E tests are automatically generated/repaired by Playwright Agents -- manual writing/editing is prohibited
- Logic tests are generated by plan-tests during planning and stored as plan artifacts
- Developers copy test files, verify Red (fail), implement, verify Green (pass)
- Verification order: Logic tests must pass first → then E2E must pass

## Test Spec Writing Rules

### 한국어 명세 + AAA 패턴 (Contract-first)

- **describe**: subject under test at the logic boundary (hook, validator, function, service, use case)
- **it**: 한국어 행동 명세 + 필요 시 제약 ID (`[C-...]`)
- **AAA**: Arrange → Act → Assert
- **관점:** 테스트는 구현을 통과시키기 위한 최소 조건이 아니라, 구현이 따라야 할 로직 계약이다

```typescript
describe('SignupPolicy', () => {
  it('[C-AUTH-001] 유효한 입력이면 회원 가입을 완료한다', async () => {
    // Arrange
    const request = { email: 'test@example.com', password: 'Password123!' };

    // Act
    const result = await subject.execute(request);

    // Assert
    expect(result.status).toBe('success');
  });
});
```

## What to Test vs. What Not to Test

### E2E targets (auto-generated by Playwright Agents)

- User flows, UI interactions, page navigation
- Data flow (API → UI → feedback), authorization checks

### Unit / Logic targets (generated by plan-tests during planning)

- API hooks (`useLogin`, `useGetUser`)
- Custom hooks (`useLoginForm`, `useDebounce`)
- Validators, mappers, service or utility functions
- State management, business logic, domain policies, use cases

### Not tested

- UI layout (E2E coverage is sufficient), styling/CSS, animations, type checking (TypeScript)

---

## Core Unit Test Rules

1. **AAA pattern required**: Every test must include Arrange/Act/Assert comments
2. **Independence**: Each test must be runnable in isolation; sharing state between tests is prohibited
3. **Spec-oriented describe/it**: Test names must clearly express behavioral requirements
4. **Defensive coverage**: Every requirement should include non-happy-path verification such as validation, boundary, or exception handling
5. **Mocking**: Use the project's established tools and mock external boundaries only

## Unit Stack Policy

- Follow the stack and test conventions already present in the repository
- If the stack is unclear, ask before generating tests
- Examples only when the repo indicates that ecosystem:
  - React / Next.js: `Vitest + Testing Library + happy-dom/jsdom`
  - NestJS: `Jest + Nest testing utilities`
  - Java / Spring: `JUnit 5 + Mockito + Spring Boot Test`

## E2E vs. Unit Decision Criteria

| Criterion | E2E (Playwright) | Unit / Logic Test |
|-----------|-------------------|-------------------|
| User flow | O | - |
| UI interaction | O | - |
| Navigation | O | - |
| Custom hook logic | - | O |
| API hook | - | O |
| Utility function | - | O |
| Business logic | - | O |
| State management | - | O |

## Test File Structure and Location

### E2E (managed by Playwright Agents)

```
specs/{domain}/{feature}.md        # Planner-generated plan
e2e/{domain}/{scenario}.spec.ts    # Generator-produced test
e2e/seed.spec.ts                   # Environment bootstrap (manually managed)
```

### Unit / Logic

- Follow the repository's local convention instead of forcing a single format
- Mirror the eventual destination path under `plans/{task-name}/tests/`

Examples:

- TypeScript repo: `src/.../__tests__/index.test.ts` or adjacent `*.spec.ts`
- Java / Spring repo: `src/test/java/.../*Test.java`
